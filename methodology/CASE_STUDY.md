# 案例：sanitary-admin 卫浴管理系统实践记录

> **声明：** 本案例是使用《多 Agent 协同开发方法论》的一个具体实践记录。你的项目技术栈和可用模型可能不同，但方法论通用。

---

## 项目背景

**项目名称：** sanitary-admin 卫浴管理系统

**技术栈：**
- 后端：Spring Boot 2.7 + MyBatis-Plus + MySQL
- 前端：Vue 3 + Element Plus + TypeScript
- 部署：Docker + Nginx

**开发目标：** 使用多 Agent 流水线自动完成模块开发、代码审查、测试验证

## 遇到的具体问题与解法

### 问题1：主 Agent 阻塞

**现象：** 在执行 mvn compile 时，主 Agent 长时间无响应

**原因：** 开发 Agent 在主线程直接执行编译命令

**解决：**
- 将所有耗时操作改为 spawn sub-agent
- 主 Agent 只负责调度和监控
- 设置任务超时时间（600 秒）

### 问题2：Sub-Agent 卡死

**现象：** 某个 Sub-Agent 状态显示 running，但 20 分钟无输出

**原因：** Prompt 中存在等待用户输入的逻辑

**解决：**
- 在所有 Prompt 中明确要求：**不询问确认，直接执行**
- 添加心跳监控，超过 10 分钟无响应自动终止
- 记录详细日志便于排查

### 问题3：Token 超限

**现象：** 审查 Agent 处理大文件时报错

**原因：** 单次请求 token 超过模型限制

**解决：**
- 将大文件拆分审查
- 限制每次审查的文件数量
- 使用更大上下文窗口的模型

### 问题4：修复循环无限

**现象：** 修复后审查仍然不通过，循环 5 次仍未解决

**原因：**
- 问题描述不够清晰
- 修复 Agent 理解有偏差

**解决：**
- 审查报告增加具体的修复建议
- 限制最大循环次数为 3
- 超过限制后人工介入

## 经验总结

### 成功经验

1. **角色分工明确**：每个 Agent 只做一件事，Prompt 优化更容易
2. **质量门控有效**：审查 + 测试 + 修复循环确保代码质量
3. **可观测性强**：状态文件、报告文件便于追踪问题
4. **模型选型合理**：根据任务特性选择不同能力的模型

### 需要改进

1. **修复循环优化**：某些问题多次迭代仍未解决，需要更精确的问题描述
2. **并行执行**：目前是串行执行，部分独立任务可以并行
3. **增量审查**：目前审查整个模块，可以只审查变更部分
4. **测试覆盖**：接口测试覆盖不全，需要补充更多场景

## 给其他项目的建议

1. **先跑通一个简单模块**，熟悉流水线流程
2. **持续优化 Prompt**，记录哪些表述效果更好
3. **建立代码规范文档**，让 Developer Agent 遵循项目风格
4. **定期回顾审查报告**，发现常见问题模式
5. **保留成功案例**，作为 Prompt 参考
